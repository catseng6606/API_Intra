# API 簡介：應用程式介面是什麼？ (What is an API?)

**API** (Application Programming Interface) 簡而言之，就是一套定義好如何溝通的 **「介面」或「合約 (Contract)」**。它定義了不同軟體元件（例如，你的網站前端和後端伺服器）之間可以互相呼叫和交換資料的規則。

---

## 🍴 實例比喻：餐廳的服務生 (The Waiter Analogy)

想像你走進一家高級餐廳準備用餐：

*   **你 (Client / 客戶端)：** 你是顧客，肚子餓了想要一份美味的餐點（取得資料）。你不需要知道廚房裡有幾位廚師、烤箱是什麼牌子，也不用親自下廚（這代表你不需要了解後端的複雜邏輯與資料庫結構）。
*   **廚房 (Server / 伺服器端)：** 這是餐廳的心臟地帶，負責實際烹飪（處理業務邏輯與資料運算）。廚房擁有所有的食材和食譜（資料庫），並知道如何把生食變成佳餚。
*   **服務生 (API / 應用程式介面)：** 服務生就是你和廚房之間的**「橋樑」或「合約」**。你不會直接衝進廚房大喊你要什麼，而是把需求告訴服務生。服務生負責將你的點餐請求（Request）送進廚房，等廚房把菜做好後，再端出來交給你（Response）。
*   **點單紙上的固定格式 (JSON / 資料交換格式)：** 為了避免溝通不良（例如你點「稍微熟一點的牛排」，廚房聽成「全熟」），你們約定好點餐必須寫在特定的點單紙上。這張紙規定了必須填寫：菜名、份數、熟度（例如：`{"meal": "steak", "doneness": "medium", "quantity": 1}`）。這就是 JSON 的角色——一種大家都看得懂、格式統一的**「共同語言」**。

**📌 總結 API 的核心價值：「隔離 (Decoupling)」**
API 讓你可以輕鬆地取得你想要的結果，而完全不需要介入後端繁瑣的運作細節。只要按照規定好的格式（JSON）透過介面（服務生）提出請求，就能拿到你想要的資料（餐點）。

## 🧩 什麼是 API？ (The Technical Definition)

從技術層面看，API 溝通時包含以下核心要素：

1.  **網域 (Domain / Base URL)：** 提供 API 服務的伺服器位址（如同餐廳的實際地址）。例如：`https://api.example.com`。
2.  **端點 (Endpoints)：** 依附在網域之下的特定路徑，代表一個具體的資源或功能（如同餐廳裡的特定點餐櫃台）。例如：`/api/users`。結合網域即為完整的 API 網址：`https://api.example.com/api/users`。
3.  **方法 (HTTP Methods)：** 用來指定你對該資源想要執行的動作。最常見的就是 `GET`（取得資料）、`POST`（建立資料）、`PUT`（更新資料）以及 `DELETE`（刪除資料）。
4.  **格式 (Format)：** 資料傳遞時必須遵循的結構，最常見的是 JSON 或 XML。

### ⚙️ 參數傳遞的大項概論 (Introduction to Parameters)

在 API 的「合約」中，我們透過不同的**載體 (Carrier)** 來傳遞資料。主要包含以下四大項：

1.  **路徑參數 (Path Parameters):** 直接嵌入在 URL 路徑中，用於定位特定資源（例如：`/users/123`）。
2.  **查詢參數 (Query Parameters):** 附加在 URL 後方（`?` 之後），用於篩選或分頁（例如：`/users?status=active`）。
3.  **請求體 (Request Body):** 放在請求的內部（通常是 JSON 格式），用於傳送複雜或大量的資料。
4.  **HTTP Header (標頭):** 用於傳遞**元數據**，例如身份證明的 Token 或資料格式的定義。
